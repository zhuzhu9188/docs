网站出了问题，最让人焦虑的不是问题本身，而是不知道从哪里开始排查。流量掉了一半，是被降权了还是竞争对手突然变强？广告有点击却没转化，是落地页的问题还是追踪代码出了错？本页面把最常见的几类故障整理成系统的排查清单，每个问题都有明确的诊断步骤，帮你像医生一样逐步缩小问题范围，找到真正的病根。

## 流量问题

<Accordion title="🔻 流量突然大幅下降">

流量突然下降是最让站长心跳加速的信号。在恐慌之前，先按照以下步骤逐一排查。

**第一步：确认数据本身是否准确**

先排除统计工具问题本身。检查 GA4 中的日期范围是否正确，看看是否有节假日或特殊事件影响了正常流量模式。对比 Google Search Console 的数据——如果 GA4 掉了但 Search Console 没变化，可能是追踪代码出了问题。

**第二步：检查 Google Search Console 手动处罚通知**

进入 Search Console → 安全和手动操作 → 手动操作，查看是否有处罚通知。如果有，页面会明确显示处罚原因（如垃圾链接、内容欺骗等）。手动处罚需要修复问题后提交重新审核申请。

**第三步：检查是否赶上算法更新**

Google 每年发布多次核心算法更新，每次都可能影响特定类型网站的排名。

<Note>
推荐工具：[Semrush Sensor](https://www.semrush.com/sensor/)（每日显示 Google 算法波动程度）和 [MozCast](https://moz.com/mozcast/)（算法变动天气预报）。如果这两个工具显示近期算法波动剧烈，你的流量下降可能是算法调整导致的。
</Note>

**第四步：检查技术问题**

这类问题经常被忽视，但影响巨大：

- `robots.txt` 是否误屏蔽了重要目录？访问 `yourdomain.com/robots.txt` 检查 `Disallow` 规则
- 页面是否被设置了 `noindex`？用 Screaming Frog 爬取网站，过滤出带 noindex 标签的页面
- `canonical` 标签是否指向了错误的 URL？重复 canonical 会导致 Google 混淆抓取权重
- 服务器是否有宕机记录？使用 [UptimeRobot](https://uptimerobot.com) 查看历史可用率

**第五步：分析竞争对手变化**

使用 Ahrefs 或 Semrush 查看主要竞争对手的流量变化。如果竞争对手流量也下降，问题可能在行业层面；如果竞争对手增长而你下降，则需要深入对比内容质量和外链差距。
</Accordion>

<Accordion title="📊 特定页面流量下降但整体正常">

当某几个页面突然失去流量，而整站数据正常时，通常是页面级别的问题。

**排查思路：**

1. 在 Search Console 中找到这些页面，查看展示次数和点击量的历史趋势——是排名下滑还是点击率下降？
2. 用无痕模式搜索目标关键词，看看当前排名位置
3. 对比当前排名第一的竞争对手页面与你的页面：内容深度、更新时间、结构化数据是否有明显差距？
4. 检查页面是否有技术变动：URL 是否有改动但未正确301重定向，页面是否被误加了 noindex

<Warning>
如果你最近修改了 URL 结构但忘记设置301重定向，旧 URL 的所有 SEO 权重都会流失。用 Screaming Frog 检查所有4xx错误页面。
</Warning>
</Accordion>

## 广告问题

<Accordion title="📉 广告点击率（CTR）异常偏低">

行业平均广告点击率因类型不同差异很大（搜索广告通常在3%-10%，展示广告在0.1%-0.5%）。如果你的点击率远低于行业均值，按以下方向排查。

**广告文案是否与搜索意图匹配**

打开 Google Ads 的搜索词报告（Search Terms Report），查看触发广告的实际搜索词。如果广告文案是"购买高端跑鞋"，但触发词是"跑鞋评测推荐"，意图不匹配会直接压低点击率。

**关键词匹配类型是否过于宽泛**

广泛匹配（Broad Match）会触发大量不相关搜索词，导致整体 CTR 下降。尝试切换到词组匹配（Phrase Match）或精确匹配（Exact Match），提高相关性。

**是否充分利用了广告扩展**

广告扩展（Ad Extensions / Assets）可以显著增加广告面积和可点击区域，提升 CTR。检查是否启用了：网站链接扩展、摘要扩展、结构化摘要、电话扩展（如适用）。

**参考竞争对手广告文案**

<Note>
使用 [Google Ads 透明度中心](https://adstransparency.google.com) 或 Semrush 的广告研究功能，查看同行业竞争对手正在运行的广告文案，找到可以学习或差异化的方向。
</Note>
</Accordion>

<Accordion title="🖱️ 广告有点击但没有转化">

有点击没转化，是比没点击更让人沮丧的问题——因为广告费已经花出去了，却没有任何回报。

**落地页与广告信息一致性检查**

从广告点击进来的用户，在落地页看到的第一屏内容是否和广告承诺的一致？标题、主视觉、优惠信息必须完全对应，形成连贯的"信息气味"。

**落地页加载速度检查**

<Note>
使用 [Google PageSpeed Insights](https://pagespeed.web.dev) 检测落地页，重点关注移动端得分。如果 LCP（最大内容绘制）超过4秒，建议优先解决速度问题——每秒延迟会导致转化率下降约7%。
</Note>

**转化追踪是否正确配置**

这是一个常被忽视的问题：可能你的转化其实发生了，只是没被正确记录。用 Google Tag Manager Preview 模式模拟完整的转化流程，验证每个步骤的事件是否正常触发。检查 Google Ads 中的转化操作状态是否显示"正在记录转化"。

**表单在移动端是否可用**

超过50%的广告点击来自手机端。在真实手机上完整走一遍转化流程：点击广告 → 进入落地页 → 填写表单 → 提交成功。特别注意：手机键盘是否遮挡表单字段、按钮是否足够大、表单字段数量是否过多。

**用户行为录制分析**

安装 Microsoft Clarity（完全免费），录制用户在落地页的真实行为。查看用户是否在某个位置停下来、反复滚动、点击非链接区域——这些都是页面存在问题的信号。
</Accordion>

## SEO 问题

<Accordion title="📈 SEO 排名停滞不前，长期卡在第5-10位">

卡在第一页后半段是最令人沮丧的处境——有展示、有少量流量，但永远进不了前三名。

**内容质量对比**

打开无痕浏览器，搜索你的目标关键词，仔细阅读排名前3的竞争对手页面：他们覆盖了哪些你没有提到的子话题？有没有独特的数据、案例或工具？内容结构（H2/H3 层级）是否比你更清晰？用户完成阅读后是否能得到更完整的答案？

**外链数量和质量对比**

<Note>
在 Ahrefs 中搜索你的页面 URL 和竞争对手的 URL，对比 Referring Domains（外部链接域名数量）和 DR（域名评分）。如果对手有50个高质量外链而你只有5个，排名差距是合理的。
</Note>

**页面技术问题排查**

- Core Web Vitals 是否达标（可在 Search Console 的"体验"报告中查看）
- 页面是否有内容重复问题（可用 Copyscape 或 Siteliner 检查）
- Internal Linking：是否有足够多的内部链接指向这个页面，帮助传递权重？

**关键词选择是否过于激进**

如果你的网站 DA（域名权威度）是20，却在竞争 DA 70+ 的大网站主导的关键词，短期内很难突破。尝试用 Ahrefs 的 Keyword Difficulty 过滤出 KD 30以下的长尾词，先积累排名信用。
</Accordion>

<Accordion title="🔍 网站在 Google 搜索中完全消失">

如果你的网站在 Google 上完全找不到（包括搜索 `site:yourdomain.com` 也没有结果），可能遇到了严重的技术或处罚问题。

**排查步骤：**

1. 搜索 `site:yourdomain.com`，查看是否有任何页面被收录
2. 进入 Search Console 检查"覆盖范围"报告，查看被排除的 URL 原因
3. 检查 `robots.txt` 是否有 `Disallow: /` 全站屏蔽
4. 检查 `<meta name="robots" content="noindex">` 是否被误加在全站模板上
5. 确认域名没有过期或 DNS 解析没有失效

<Warning>
如果网站是新上线的，Google 完整收录通常需要2-4周。可以在 Search Console 中手动提交 Sitemap 加速收录流程。
</Warning>
</Accordion>

## 速度问题

<Accordion title="⚡ 网站速度慢，PageSpeed 得分低">

页面速度直接影响用户体验、SEO 排名和广告质量得分。以下是系统化的诊断和修复流程。

**第一步：诊断具体瓶颈**

<Note>
使用 [Google PageSpeed Insights](https://pagespeed.web.dev) 获取具体的性能报告，重点关注三个核心指标：LCP（最大内容绘制，目标＜2.5秒）、FID/INP（交互响应，目标＜200ms）、CLS（累积布局偏移，目标＜0.1）。报告会列出具体的优化建议，按影响程度排序。
</Note>

**图片问题（最常见的速度杀手）**

- 图片是否转换为 WebP 格式？同等质量下比 JPEG 小约30%
- 是否设置了懒加载（`loading="lazy"`）？首屏以外的图片不应该立即加载
- 是否指定了图片的 `width` 和 `height` 属性？缺失会导致 CLS 问题

**未启用浏览器缓存**

检查你的主机或CDN配置，确保静态资源（CSS、JS、图片）设置了合理的缓存时间（建议1年以上）。Cloudflare 的免费方案已经自动处理大部分缓存配置。

**主机配置不足**

共享主机在流量集中时会响应变慢。用 [TTFB Checker](https://www.ttfb.io) 检查首字节时间（TTFB）。如果超过600ms，可能需要升级主机方案或迁移到更好的服务器（推荐 SiteGround 或阿里云 ECS）。

**JavaScript 阻塞渲染**

过多的第三方脚本（统计工具、广告代码、聊天插件）会阻塞页面渲染。通过 Google Tag Manager 统一管理所有脚本，并使用异步加载（`async`/`defer`）方式引入 JS 文件。

**WordPress 特定优化**

如果你使用 WordPress，额外检查：是否安装了太多插件（超过20个就需要审查）、是否使用了页面缓存插件（WP Rocket 或 W3 Total Cache）、数据库是否需要清理优化。
</Accordion>
