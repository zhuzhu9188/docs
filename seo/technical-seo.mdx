技术 SEO 是 SEO 工作中最容易被内容创作者忽视、却最可能成为排名瓶颈的部分。你可以写出十篇优质文章，但如果 robots.txt 配置错误、页面加载时间超过 5 秒，或者重复内容问题没有处理好，Google 爬虫会把大量资源浪费在错误的地方，你的内容也无法得到应有的排名。这篇指南带你系统梳理技术 SEO 的核心工作，逐项排查和修复影响你网站表现的底层问题。

## 爬取与索引配置

### robots.txt

`robots.txt` 是放置在网站根目录的文本文件（如 `https://example.com/robots.txt`），告诉搜索引擎爬虫哪些页面可以爬取、哪些不可以。

```txt
# 允许所有爬虫访问所有内容（默认情况）
User-agent: *
Allow: /

# 禁止所有爬虫访问管理后台
User-agent: *
Disallow: /admin/
Disallow: /wp-admin/

# 禁止访问搜索结果页（避免无限爬取）
Disallow: /search?

# 禁止 GPTBot（OpenAI 爬虫）访问内容
User-agent: GPTBot
Disallow: /

# 指向 Sitemap 位置
Sitemap: https://example.com/sitemap.xml
```

<Warning>
  `robots.txt` 是建议性的，不是强制性的。它告诉爬虫"不要爬取这里"，但不会阻止页面出现在搜索结果中（如果有其他页面链接到被禁止的页面）。如果你想彻底防止某页面出现在搜索结果中，应使用 `noindex` 标签，而不仅仅依赖 robots.txt。
</Warning>

### noindex 标签的正确使用场景

在页面 `<head>` 中添加以下标签，可以告诉 Google 不要将该页面纳入搜索索引：

```html
<meta name="robots" content="noindex, nofollow">
```

**应该使用 noindex 的场景：**
- 用户登录后才能访问的内容页
- 分页页面（如 `?page=2`，避免内容稀薄页面消耗爬取预算）
- 标签页、存档页（内容聚合度低）
- 感谢页、结账成功页
- 开发测试环境

**不应该使用 noindex 的场景：**
- 任何你想出现在搜索结果中的页面（显而易见，但误操作确实发生过）

<Note>
  在 WordPress 中，新站发布时有一个"阻止搜索引擎索引"的勾选项，很多人在开发阶段勾上后，上线时忘记取消——这是导致"为什么 Google 不收录我的网站"的最常见原因之一。上线前务必检查这个设置。
</Note>

## Sitemap.xml：帮助 Google 高效发现页面

Sitemap 是一个 XML 文件，列出了你网站的所有重要页面及其更新时间，帮助 Google 更高效地发现和优先爬取你的内容。

```xml
<?xml version="1.0" encoding="UTF-8"?>
<urlset xmlns="http://www.sitemaps.org/schemas/sitemap/0.9">
  <url>
    <loc>https://example.com/</loc>
    <lastmod>2024-06-01</lastmod>
    <changefreq>weekly</changefreq>
    <priority>1.0</priority>
  </url>
  <url>
    <loc>https://example.com/seo/fundamentals/</loc>
    <lastmod>2024-05-20</lastmod>
    <changefreq>monthly</changefreq>
    <priority>0.8</priority>
  </url>
</urlset>
```

**生成 Sitemap 的常用方法：**

<Tabs>
  <Tab title="WordPress">
    安装 **Yoast SEO** 或 **Rank Math** 插件后，Sitemap 会自动生成，访问地址通常为 `yourdomain.com/sitemap_index.xml`。无需手动维护，插件会在新页面发布时自动更新。
  </Tab>
  <Tab title="静态网站">
    如果使用 Next.js、Gatsby 等框架，可通过对应插件自动生成。如使用纯静态 HTML，可使用 [XML Sitemaps Generator](https://www.xml-sitemaps.com/)（免费在线工具）生成，但需要在每次更新网站后重新生成并上传。
  </Tab>
  <Tab title="手动提交">
    生成 Sitemap 后，登录 **Google Search Console** → 左侧菜单"站点地图" → 输入 Sitemap URL → 点击"提交"。提交后 Google 会开始处理，通常在数天内完成初次爬取。
  </Tab>
</Tabs>

## 页面速度与 Core Web Vitals

Core Web Vitals 是 Google 用于衡量用户实际页面体验的一组指标，自 2021 年起成为正式排名因素。不达标不一定导致排名大幅下降，但在竞争对手内容质量相近时，页面体验差的网站会处于劣势。

**三大核心指标：**

| 指标 | 全称 | 衡量内容 | 良好标准 |
|------|------|---------|---------|
| LCP | Largest Contentful Paint | 最大内容元素加载时间 | ≤ 2.5 秒 |
| INP | Interaction to Next Paint | 交互响应速度（替代了 FID）| ≤ 200 毫秒 |
| CLS | Cumulative Layout Shift | 页面布局稳定性 | ≤ 0.1 |

<Steps>
  <Step title="测量当前状态">
    使用 [Google PageSpeed Insights](https://pagespeed.web.dev/) 输入你的页面 URL，获取 LCP、INP、CLS 的当前数值。注意区分"实验室数据"（模拟）和"真实用户数据"（Field Data）——Google 排名算法基于真实用户数据。

    也可以在 Google Search Console 的"Core Web Vitals"报告中查看按页面分组的整体状况。
  </Step>
  <Step title="优化 LCP（最大内容加载）">
    LCP 不达标通常因为：首屏大图未优化、服务器响应慢、阻塞渲染的 CSS/JS。

    主要优化手段：
    - 首屏主图使用 `<link rel="preload">` 预加载
    - 将图片转换为 WebP 格式并压缩
    - 开启服务器端缓存和 CDN（内容分发网络）
    - 将不影响首屏的 JS 设置为 `defer` 或 `async` 加载
  </Step>
  <Step title="优化 CLS（布局偏移）">
    CLS 高通常是因为图片、广告、嵌入内容没有预留尺寸空间，加载时把其他内容挤开。

    主要优化手段：
    - 为所有图片和视频标签明确指定 `width` 和 `height` 属性
    - 为广告位预留固定高度占位符
    - 避免在页面加载后动态插入内容（尤其是在首屏区域）
    - 字体加载使用 `font-display: swap`，避免字体切换时的布局跳动
  </Step>
  <Step title="优化 INP（交互响应）">
    INP 衡量用户点击、输入等交互后页面的响应速度。主要优化手段：
    - 减少主线程 JavaScript 执行时间
    - 拆分长任务（超过 50ms 的 JS 任务）
    - 延迟加载非关键第三方脚本（如客服聊天插件、热力图工具）
  </Step>
  <Step title="持续监控">
    将 Core Web Vitals 指标加入定期检查清单。每次网站改版或新增第三方脚本后，重新测量一次，避免性能回退。
  </Step>
</Steps>

## 移动端适配：移动优先索引

自 2023 年起，Google 已全面切换到**移动优先索引**（Mobile-First Indexing），这意味着 Google 主要以移动版页面作为索引和排名的依据。如果你的移动端体验差于桌面端，排名会受到影响。

**检查移动端友好性：**
- 使用 [Google Mobile-Friendly Test](https://search.google.com/test/mobile-friendly) 测试你的页面
- 在 Google Search Console 的"移动易用性"报告中查看批量问题

**常见移动端问题：**
- 文字太小需要缩放才能阅读（建议正文字号 ≥ 16px）
- 可点击元素（按钮、链接）过于密集，容易误触（建议间距 ≥ 8px）
- 内容宽度超出屏幕，需要横向滚动
- 移动端加载了不必要的桌面端大图

## HTTPS 与安全信号

Google 自 2014 年起将 HTTPS 列为正式排名信号。今天，HTTPS 已是基本要求而非加分项——不使用 HTTPS 的网站会被浏览器标记为"不安全"，直接影响用户信任度和转化率。

**启用 HTTPS 的步骤：**
1. 从主机商或 [Let's Encrypt](https://letsencrypt.org/)（免费）获取 SSL 证书
2. 将所有 HTTP 请求通过 301 重定向指向 HTTPS 版本
3. 更新内部链接和 Sitemap 中的 URL 为 HTTPS
4. 在 Google Search Console 添加并验证 HTTPS 版本的域名

## 重复内容处理

重复内容会分散 Google 的排名信号，导致多个相似页面互相竞争，所有版本的排名都不理想。

### Canonical 标签

当同一内容存在于多个 URL 时，使用 canonical 标签告诉 Google 哪个是"正版"页面：

```html
<!-- 在重复页面的 <head> 中添加 -->
<link rel="canonical" href="https://example.com/seo/on-page-optimization/" />
```

**常见需要 canonical 的场景：**
- 带参数的 URL（`/page?sort=price`、`/page?ref=newsletter`）
- HTTP 和 HTTPS 并存
- 有无 www 的两个版本（`www.example.com` vs `example.com`）
- 分页页面中的内容重复

### 301 重定向

当你永久移动或删除一个页面时，使用 301 重定向将旧 URL 的权重传递给新 URL。在 Apache 服务器的 `.htaccess` 文件中配置示例：

```apache
# 单条 URL 重定向
Redirect 301 /old-page/ https://example.com/new-page/

# 将整个 HTTP 站点重定向到 HTTPS
RewriteEngine On
RewriteCond %{HTTPS} off
RewriteRule ^(.*)$ https://%{HTTP_HOST}%{REQUEST_URI} [L,R=301]

# 去掉 www
RewriteCond %{HTTP_HOST} ^www\.example\.com [NC]
RewriteRule ^(.*)$ https://example.com/$1 [L,R=301]
```

<Warning>
  避免使用 302 临时重定向来处理永久性的 URL 变更，302 不会传递原页面的 SEO 权重，导致你"白白"失去积累的排名权重。
</Warning>

## 结构化数据验证

添加 Schema 结构化数据后，使用 **Google Rich Results Test** 验证代码是否有效：

<Steps>
  <Step title="测试单个页面">
    访问 [search.google.com/test/rich-results](https://search.google.com/test/rich-results)，输入页面 URL 或直接粘贴 HTML 代码，查看 Google 能识别到哪些结构化数据类型，以及是否有错误或警告。
  </Step>
  <Step title="在 Search Console 批量检查">
    Google Search Console 的"增强功能"板块会显示你网站中各类结构化数据（文章、FAQ、面包屑等）的总体有效页面数和错误数，方便批量排查问题。
  </Step>
  <Step title="修复常见错误">
    常见错误包括：必填字段缺失（如 Article 缺少 `datePublished`）、字段格式不正确（日期格式应为 ISO 8601，如 `2024-06-01`）、标记内容与页面实际内容不匹配。
  </Step>
</Steps>

## 使用 Screaming Frog 诊断常见技术 SEO 问题

Screaming Frog SEO Spider 是一款桌面端爬虫工具，可以模拟 Google 爬虫扫描你的网站，发现技术 SEO 问题。**免费版支持爬取最多 500 个 URL**，对中小型网站完全够用。

<Accordion title="404 错误页面">
  **问题**：用户或爬虫访问已删除或移动的 URL，出现"页面不存在"。

  **检查方法**：在 Screaming Frog 中筛选响应码为 404 的 URL，查看哪些页面正在链接到这些 404 页面。

  **修复方案**：设置 301 重定向，将 404 URL 指向最相关的现有页面；或如果该页面确实已废弃，更新所有指向它的内链。
</Accordion>

<Accordion title="重定向链（Redirect Chains）">
  **问题**：URL A → URL B → URL C 的多级重定向链会浪费爬取预算，并导致权重传递损耗。

  **检查方法**：Screaming Frog 的"重定向"报告会显示所有重定向链。

  **修复方案**：将重定向链合并为直接的一步跳转（A 直接 301 到 C）。
</Accordion>

<Accordion title="缺失或重复的 Title/Meta">
  **问题**：多个页面共用相同 Title Tag，或某些页面缺少 Title/Meta Description。

  **检查方法**：在 Screaming Frog 的"页面标题"和"元描述"标签下，可以直接筛选"缺失"、"重复"、"超长"等问题。

  **修复方案**：逐一为缺失或重复的页面编写唯一的 Title 和 Meta Description。
</Accordion>

<Accordion title="过深的页面层级">
  **问题**：距离首页点击层数超过 3-4 层的页面，爬虫访问频率会降低，用户也难以通过导航找到。

  **检查方法**：Screaming Frog 的"层级"（Crawl Depth）报告显示每个页面距离首页的点击深度。

  **修复方案**：通过内链优化（在重要内容页之间添加相互链接、在首页或分类页添加链接）缩短深层页面与首页的距离。
</Accordion>

<Note>
  技术 SEO 问题的优先级排序建议：**先解决阻止索引的问题**（如误配置的 noindex、robots.txt 封锁）→ **再处理404和重定向链**（影响爬取效率）→ **然后优化页面速度**（影响用户体验和 Core Web Vitals）→ **最后完善结构化数据**（增强搜索结果展示）。不要试图一次性解决所有问题，按影响程度有序推进效率最高。
</Note>
